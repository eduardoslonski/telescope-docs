title: "Reward Design"
---

Reward design determines what your model learns. The reward function is the primary training signal in RL — get it right and the model improves steadily; get it wrong and you'll see stagnation, reward hacking, or collapse.

This page covers reward patterns, best practices, and common pitfalls. For the mechanics of implementing `compute_reward`, see [Environments](/training/environments).

## How rewards are used

Your `compute_reward` returns a `RewardResult`:

```python
RewardResult(
    total_reward=1.0,                            # scalar used for training
    sample_metrics={"format": 0.5, "correct": 1.0},  # component breakdown for logging
    golden_answers={"answer": "42"},             # ground truth for the UI
)
```

`total_reward` feeds into advantage computation: within each prompt group, rewards are normalized (mean-centered and divided by standard deviation) to produce advantages. The advantage determines how much the model's gradient pushes toward or away from each completion.

`sample_metrics` breaks down the reward into named components. These are logged to W&B and shown in the Telescope UI — essential for understanding what's driving learning.

## Reward patterns

### Binary correctness

The simplest pattern: 1.0 if correct, 0.0 if not. Works well for tasks with clear right/wrong answers.

```python
def compute_reward(self, completion, sample, eos_token=""):
    if eos_token and completion.endswith(eos_token):
        completion = completion[:-len(eos_token)]

    predicted = extract_boxed_answer(strip_think_tags(completion))
    correct = verify_math_answer(predicted, sample.answer)

    reward = 1.0 if correct else 0.0
    return RewardResult(
        total_reward=reward,
        sample_metrics={"correct": reward},
        golden_answers={"correct": sample.answer},
    )
```

Binary rewards are effective when:
- The task has verifiable answers (math, code with test suites)
- The model already gets some samples right (so there's signal to learn from)
- `group_size` is large enough that groups contain both correct and incorrect samples

### Multi-component rewards

Combine multiple reward signals additively. This is useful when you want the model to learn both *what* to produce and *how* to produce it.

The Countdown environment uses two components — format and correctness:

```python
def compute_format_reward(completion: str, eos_token: str) -> float:
    """
    1.0 if <think>...</think>\n<answer>...</answer> with valid equation chars.
    0.5 if tags are present but answer has unexpected characters.
    0.0 if format is wrong.
    """
    # Check for proper <think>...</think>\n<answer>...</answer> structure
    # and validate the answer only contains numbers, operators, parentheses
    ...


def compute_equation_reward(completion: str, nums: list[int], target: int) -> float:
    """1.0 if equation uses exactly the given numbers and evaluates to target."""
    match = re.search(r"<answer>(.*?)<\/answer>", completion)
    if match is None:
        return 0.0

    equation = match.group(1).strip()
    used_numbers = [int(n) for n in re.findall(r"\d+", equation)]
    if sorted(used_numbers) != sorted(nums):
        return 0.0

    result = eval(equation, {"__builtins__": None}, {})
    return 1.0 if abs(float(result) - float(target)) < 1e-5 else 0.0
```

The environment's `compute_reward` combines them:

```python
def compute_reward(self, completion, sample, eos_token=""):
    format_score = compute_format_reward(completion, eos_token)
    equation_score = compute_equation_reward(completion, sample.metadata["nums"], sample.metadata["target"])

    return RewardResult(
        total_reward=format_score + equation_score,
        sample_metrics={"format_reward": format_score, "equation_reward": equation_score},
        golden_answers={"target": str(sample.metadata["target"])},
    )
```

The format reward teaches the model to use `<think>` and `<answer>` tags even before it solves problems correctly. The 0.0/0.5/1.0 graduated scale provides partial credit — the model gets signal for getting the format right even if the equation is wrong.

### Tool-use rewards

For tool environments, you can reward not just correctness but how the model uses tools. For example, a tool-using environment might combine correctness, efficiency, and format:

```python
def compute_reward(self, state, eos_token=""):
    expected = state.sample.metadata["expected_answer"]
    model_answer = self._extract_final_answer(state)
    tool_metrics = self.get_tool_metrics(state)
    total_calls = tool_metrics.get("total_tool_calls", 0)

    # Correctness requires both right answer AND tool use
    answer_matches = abs(float(model_answer) - float(expected)) < 1e-6
    correctness = 1.0 if answer_matches and total_calls >= 1 else 0.0

    # Efficiency bonus for minimal tool use
    if 1 <= total_calls <= 2:
        efficiency = 0.2
    elif total_calls > 2:
        efficiency = max(0.0, 0.2 - 0.05 * (total_calls - 2))
    else:
        efficiency = 0.0

    # Format bonus for providing a structured answer
    format_reward = 0.1 if model_answer is not None else 0.0

    return RewardResult(
        total_reward=correctness + efficiency + format_reward,
        sample_metrics={"correctness": correctness, "efficiency": efficiency, "format": format_reward, **tool_metrics},
        golden_answers={"expected": str(expected), "model_answer": model_answer or ""},
    )
```

Requiring at least one tool call for correctness prevents the model from bypassing tools and guessing answers directly.

### Process rewards (multi-turn)

For multi-turn tasks, you can reward the quality of intermediate steps, not just the final outcome. For example, a multi-turn environment might track both end-to-end correctness and process quality:

```python
def compute_reward(self, state, eos_token=""):
    # End-to-end: did the model reach the correct final state?
    e2e = compute_end_to_end_score(state)

    # Process: how efficient was the model's trajectory?
    process = compute_process_score(state)

    return RewardResult(
        total_reward=e2e,  # end-to-end is the sole training signal
        sample_metrics={
            "end_to_end_reward": e2e,
            "process_reward": process,      # tracked for analysis, not used in loss
        },
    )
```

Here the process score is logged in `sample_metrics` for monitoring but doesn't affect `total_reward`. This is a common pattern — track metrics for analysis without coupling them to the training signal.

## Best practices

### Avoid sparse rewards

If most completions score 0.0, the model receives almost no gradient signal. When all samples in a group get the same reward, the advantage is zero — there's nothing to learn from.

Signs of sparse rewards:
- `discard_group_zero_advantage` (enabled by default) filters out most groups
- Training metrics show very small effective batch sizes
- Reward mean stays flat over many steps

Fixes:
- **Add partial credit.** Format rewards, intermediate correctness, tool usage bonuses — anything that differentiates completions even when the final answer is wrong.
- **Increase `group_size`.** More samples per prompt means more variance in the group, making it more likely that at least some completions differ.
- **Start with an easier task.** If the model can't solve any samples, RL has nothing to amplify. Consider using a smaller dataset of easier problems first.

### Use format rewards

Teaching the model to produce structured output (think tags, boxed answers, XML responses) is often worth a small reward component. The graduated pattern works well:

- **0.0** — wrong format entirely
- **0.5** — right structure but minor issues (unexpected characters, missing closing tag)
- **1.0** — correct format

This gives the model an easy early win and a consistent learning signal while it's still learning the harder parts of the task.

### Normalize reward scales across environments

When training on multiple environments with different reward ranges, set `reward_min` and `reward_max` per environment:

```yaml
environments:
  - name: "countdown"
    weight: 0.5
    reward_min: 0.0
    reward_max: 2.0    # format (0-1) + equation (0-1)
  - name: "hendrycks_math"
    weight: 0.5
    reward_min: 0.0
    reward_max: 1.0    # binary correctness
```

Without normalization, environments with larger reward ranges produce larger advantages and dominate the training signal.

### Declare metric ranges

Override `metrics_ranges` in your environment to declare the expected min/max for each `sample_metric`. The Telescope UI uses these to display normalized charts:

```python
class MyEnvironment(SingleTurnEnvironment):
    metrics_ranges = {
        "format": {"min": 0.0, "max": 1.0},
        "correctness": {"min": 0.0, "max": 1.0},
    }
```

## Common pitfalls

### Reward hacking

The model finds shortcuts that maximize reward without solving the task. For example:
- Memorizing answer patterns instead of reasoning
- Producing the right format with wrong content
- In tool environments, calling the same tool repeatedly to trigger efficiency bonuses

**Mitigation:** Make reward components complementary. Require both correctness and format. Use `sample_metrics` to monitor each component separately — if format score rises while correctness stays flat, the model may be gaming the format reward.

### Conflicting reward components

If components reward opposing behaviors, the model oscillates or converges to a compromise that's poor at both. For example, rewarding both brevity and thoroughness.

**Mitigation:** Ensure components are complementary, not conflicting. If you need to balance competing objectives, use one as the primary training signal and track the other in `sample_metrics` only.

### All-or-nothing with small groups

Binary rewards with `group_size: 4` means many groups will be all-correct or all-incorrect. With zero advantage, those groups contribute nothing to training.

**Mitigation:** Either increase `group_size` (8-16 is common), add partial credit to the reward, or both.

### Overweighting format vs substance

A format reward of 1.0 and correctness reward of 1.0 gives equal weight to both. If format is much easier, the model will focus on format because the expected improvement from format is higher.

**Mitigation:** Scale format rewards smaller than correctness (e.g., 0.1-0.5 range for format vs 1.0 for correctness), so the gradient is dominated by the harder, more important signal.
